LionAGI Core Concepts and IDs¶

The classes and interfaces described below are at the heart of LionAGI’s infrastructure. They define abstract behaviors for managers, observers, and message-oriented entities, as well as the ID system (unique UUID-based identifiers) and the Element base class. These models are essential building blocks for higher-level features like logging, action management, and graph structures.

Abstract Concepts (_concepts.py)¶

These abstract base classes (ABCs) describe fundamental “roles” or “capabilities” that a LionAGI component can fulfill. They do not provide direct implementation details, but establishing them ensures that different parts of the system communicate consistently and can be extended more reliably.

Observer¶

class lionagi.protocols._concepts.Observer¶

An abstract base (marker or interface) for all observer classes in LionAGI. Observers may be notified of events or changes in the system. Concrete classes that inherit Observer typically override or implement methods to react to these events (e.g., for logging or custom state updates).

Manager¶

class lionagi.protocols._concepts.Manager¶

Inherits from: Observer

A “manager” is a registry or controller that administers certain objects in the system. For instance, ActionManager manages callable tools, while a LogManager might manage log entries. By inheriting from Manager, a class signifies that it orchestrates or keeps track of a set of resources within LionAGI.

Relational¶

class lionagi.protocols._concepts.Relational¶

Inherits from: None (abstract base)

Used for graph-connectable objects. Classes implementing Relational can be part of a graph data structure (e.g., nodes in a knowledge graph). Typically, such classes store references to edges or neighboring elements.

Sendable¶

class lionagi.protocols._concepts.Sendable¶

An abstract base for message-oriented capabilities. A “sendable” entity must track a sender identity and potentially a recipient, enabling the system to pass or route messages. Often used in queue-based or event-driven scenarios for multi-agent communication.

Observable¶

class lionagi.protocols._concepts.Observable¶

Purpose: Denotes an object that has an id field of a known type (usually a UUID). The object is thus “observable,” allowing references or logs to identify it unambiguously. Many core LionAGI classes implement Observable so they can be stored or indexed by ID.

Communicatable¶

class lionagi.protocols._concepts.Communicatable(Observable)¶

Combines an id (from Observable) with mailbox-style communication. A “communicatable” class can have:

  • A mailbox or channel for inbound messages.

  • A send() method to emit messages to other recipients.

This underlies multi-agent or multi-branch messaging in LionAGI.

Condition¶

class lionagi.protocols._concepts.Condition¶

An abstract base specifying a single async method:

async def apply(self, *args, **kwargs) -> bool:
    ...

Typically used for permission or constraint checks. For example, EdgeCondition determines if an edge can be traversed. This method can do I/O or other async operations to decide True/False.

Collective¶

class lionagi.protocols._concepts.Collective(Generic[E])¶

A base for collections of items. Must define:

  • include(item)()

  • exclude(item)()

Subclasses, like Pile, store items with concurrency support or type constraints.

Ordering¶

class lionagi.protocols._concepts.Ordering(Generic[E])¶

Similar to Collective, but emphasizes a strict sequence of items. Must define how items are inserted or removed while preserving an order. E.g., Progression that keeps an ordered list of IDs.

ID System and Element (element.py)¶

ID System and Element (element.py¶

LionAGI uses a UUIDv4-based scheme to ensure each object is uniquely identifiable. These classes unify that approach, providing hashing/equality logic and a base class called Element that includes:

  • A unique ID field,

  • A creation timestamp,

  • A metadata dictionary for user-defined data.

IDType¶

class lionagi.protocols.generic.element.IDType¶

A lightweight wrapper around a version-4 UUID. It enforces:

  • Validation of input strings to confirm valid v4 UUIDs.

  • Creation of random new UUIDs with create().

  • Simple equality/hashing so that ID objects can be dict keys or set members.

ID¶

class lionagi.protocols.generic.element.ID(Generic[E])¶

A utility class that helps convert various inputs (string, Element, etc.) into a validated IDType. - get_id(item) -> IDType(): Create or verify a valid ID. - is_id(item) -> bool(): Quickly check if item is recognized as an ID.

Element¶

class lionagi.protocols.generic.element.Element¶

Inherits from: pydantic.BaseModel

A base class for any LionAGI object needing:

  • A unique id (UUID v4).

  • A created_at timestamp (float, Unix epoch).

  • A metadata dictionary for custom fields.

Also provides:

  • to_dict() for easy serialization,

  • Overridden __eq__ and __hash__ comparing objects by their id.

Example:

from lionagi.protocols.generic.element import Element

class MyElement(Element):
    pass

e = MyElement()
print(e.id)        # => IDType
print(e.metadata)  # => {}
# Now e can be placed into a dictionary keyed by e.id

Summary¶

These abstract concepts and ID-based data models form the core of LionAGI:

  • “Roles” like Observer, Manager, Relational, etc., guide how classes are shaped and how they interact (managing tools, sending messages, etc.).

  • Conditions for controlling certain transitions or checks.

  • The ID architecture ensures a universal reference scheme across objects in logs, graphs, or concurrency managers.

  • Element standardizes an ID, timestamp, and metadata for every signficant LionAGI component.
